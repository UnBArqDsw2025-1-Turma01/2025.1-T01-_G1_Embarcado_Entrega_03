# 3.2. Módulo Padrões de Projeto GoFs Estruturais

## Padrões Estruturais

Ao modelar um sistema orientado a objetos, especialmente com múltiplas classes interconectadas, um dos principais desafios está em **organizar e estruturar as relações entre objetos e classes**, de forma que o sistema permaneça flexível, reutilizável e de fácil manutenção. Dentre eles:

1. **Composite**: O padrão Composite tem como principal objetivo permitir que objetos sejam organizados em estruturas hierárquicas de árvore, onde objetos individuais e composições desses objetos são tratados de maneira uniforme.

2. **Adapter**:  Em sistemas que precisam integrar bibliotecas externas ou componentes legados, sem comprometer a arquitetura interna. O Adapter converte a interface de uma classe existente para uma interface esperada pelo cliente, promovendo reuso, desacoplamento e isolamento de dependências externas.

---

Inicialmente, considerou-se a criação de condicionais dentro da classe Notificacao, que verificariam o tipo de alerta e executariam manualmente o envio por cada canal necessário. No entanto, essa abordagem tornaria a classe Notificacao altamente acoplada às implementações de envio e dificultaria a manutenção e evolução do sistema, especialmente na inclusão de novos canais no futuro.

Para resolver esse problema, foi adotado o padrão Composite, por meio da implementação da classe EnvioComposto. Essa classe implementa a interface IEnvio, assim como as implementações concretas EnvioEmail, EnvioSMS e EnvioPush, mas, diferentemente delas, agrega múltiplas estratégias de envio em uma estrutura unificada.

---

No decorrer de discussões , foi deciditoinserir lógica condicional no código cliente para lidar com cada tipo específico de sensor, como Giroscopio ou Acelerometro. Porém, essa abordagem introduziria alto acoplamento, complexidade e tornaria o sistema difícil de escalar com novos sensores. Como já até explicado em sala de aula do exemplo dos vários caminhos dado um local e destino, preencher de if-else seria algo "não elegante".

Para resolver essa limitação, optou-se pela aplicação do padrão Adapter, por meio da criação da interface ISensorAdapter. Essa interface define os métodos esperados pelo sistema — como getDeltaGrau() e getDeltaTempo() — e serve como contrato único para qualquer sensor utilizado na análise.

Cada tipo de sensor real (como Giroscopio e Acelerometro) é então encapsulado em sua respectiva classe adaptadora (GiroscopioAdapter, AcelerometroAdapter), que implementa ISensorAdapter e traduz os dados e comportamentos do sensor original para o formato esperado pelo sistema.

O componente TargetMonitoramento agora trabalha diretamente com objetos do tipo ISensorAdapter, sem se preocupar com os detalhes de implementação dos sensores reais. Isso garante flexibilidade, facilidade de manutenção e prepara o sistema para adicionar novos sensores futuramente, bastando criar novos adapters compatíveis com a interface.
